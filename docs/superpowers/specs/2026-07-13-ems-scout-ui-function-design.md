# EMS Scout 功能与 UI 设计规格

日期：2026-07-13

状态：设计已批准，待书面规格复核

范围：原生 WinUI 3 应用的信息架构、核心工作流、页面交互和功能优化

## 1. 背景

EMS Scout 的核心任务是采集 6 栋楼的空调卡片数据，写入 SQLite，供运维人员筛选、审计并导出当前筛选 Excel。原生应用已经覆盖总览、采集、数据管理、分组、日期计划、审计、设置和诊断，但页面数量和能力密度快速增长，出现以下问题：

- 8 个同级导航项混合日常操作、业务配置和工程诊断。
- `AreasPage.xaml` 同时承担分组、楼层目录、成员、日期入口、关注规则和异常事件，单页职责过多。
- 数据批次选择在总览、数据管理和审计页重复出现，用户容易误判当前数据口径。
- 自定义分组不能直接作为数据管理筛选和 Excel 导出口径，分组维护与设备检查之间存在断点。
- 基础质量、实时点位、实时对账、计划偏差、关注事件和异常批次分散在不同页面，缺少统一待处理入口。
- 数据管理筛选功能完整，但缺少快捷问题筛选、设备详情、保存视图和历史变化对比。
- 当前 Debug 启动入口在审查环境中弹出 .NET 运行时错误，未能完成真实运行态 UI 巡检。安装、启动和 MSIX 生命周期仍是 UI 验收的 P0 前置门。

本规格不把上述现状描述为已实现功能。它定义获批的目标设计和后续验收边界。

## 2. 用户与设计目标

### 2.1 主要用户

主要用户是日常值班和巡检人员，核心诉求是快速完成采集、发现异常、定位设备并导出结果。

次要用户是数据管理员，负责分组、日期计划、关注规则和数据口径维护。

现场实施工程师使用采集高级参数、环境检查和诊断能力。这些能力保留，但不得干扰日常操作。

### 2.2 成功标准

- 用户在工作台首屏可以判断数据是否新鲜、是否存在阻断问题以及下一步操作。
- 从任意问题项到准确设备筛选结果不超过一次页面跳转。
- 当前数据与历史批次始终可区分，历史批次不可被误编辑或误导出。
- 分组成员可以直接查看、筛选并通过唯一导出路径导出 Excel。
- 采集任务在准备、运行和完成三个阶段提供明确状态与下一步。
- 页面在常用分辨率、高 DPI、浅色和深色主题下不发生内容重叠或关键操作不可达。
- 安装包能在目标 Windows 环境完成安装、启动、升级和卸载，才可声明 UI 验收完成。

## 3. 三轮审查结论

### 3.1 第一轮：现状与信息架构

当前原生应用能力充分，但顶层导航过多，页面职责边界不清。数据管理的“筛选结果即 Excel 导出口径”是最稳定的产品主线，应成为跨页工作流的锚点。

### 3.2 第二轮：路线比较

审查比较了三条路线：

1. 原地优化：保留 8 个顶层页面，只改善布局和反馈。风险低，但无法解决导航和跨页断点。
2. 任务流重组：按工作台、采集、设备数据、规则与计划、审计组织主流程。改动适中，最符合日常运维流程。
3. 角色工作区：提供日常运维、数据治理和工程诊断三种工作区。能力强，但会引入角色切换和功能发现成本。

批准采用第 2 条路线。当前用户规模和权限模型不足以支持角色工作区，暂不引入。

### 3.3 第三轮：页面与功能

第三轮确认了统一数据上下文、统一待处理队列、核心三页布局、规则与审计重组、辅助页面收敛以及 P0/P1/P2 优先级。详细要求见后续章节。

## 4. 产品信息架构

### 4.1 主导航

主导航收敛为：

1. 工作台
2. 采集
3. 设备数据
4. 规则与计划
5. 审计

系统设置和诊断进入 `NavigationView.FooterMenuItems`，作为工具入口，不与日常任务竞争。

### 4.2 页面职责

| 页面 | 唯一主要职责 |
|---|---|
| 工作台 | 判断当前状态、呈现优先问题和发起下一步 |
| 采集 | 运行前检查、选择范围、执行采集和查看本次结果 |
| 设备数据 | 筛选、检查设备详情、比较历史和导出 Excel |
| 规则与计划 | 定义分组、成员、日期计划和关注规则 |
| 审计 | 发现、解释和处理质量、实时、计划及历史问题 |
| 系统设置 | 管理连接、目录、采集默认值和界面偏好 |
| 诊断 | 检查运行环境、路径、日志和应用状态 |

页面之间通过深链接共享上下文，不在多个页面重复实现相同业务能力。

## 5. 共享应用能力

### 5.1 数据上下文栏

现有 `DataContextService` 继续作为唯一数据批次状态源。工作台、设备数据和审计在统一位置展示数据上下文栏：

- 当前数据标记为“当前 · 可操作”。这里的可操作指允许执行分组、问题状态等业务写操作，不允许直接修改设备采集值。
- 历史批次标记为“只读”。
- 切换批次后，三个数据消费页面同步刷新。
- 采集和规则与计划始终作用于当前库，不继承历史批次。
- 历史批次禁止导出、修改分组、修改关注状态和执行其他写操作。

### 5.2 调查导航上下文

跨页导航必须能够携带：

- 批次 ID
- 楼栋、座号、楼层、子区和页面
- 设备名或稳定设备身份
- 分组 ID
- 问题类型和问题 ID
- 来源页面和返回目标

数据页必须明确显示由跳转带入的筛选条件，并提供返回原问题的路径。导航参数不能只传展示文本；设备定位优先使用稳定身份，展示字段仅作为兼容回退。

### 5.3 待处理队列

新增应用层待处理队列，统一聚合：

- 基础质量问题
- 实时点位异常
- 实时对账差异
- 日期计划偏差
- 关注设备事件
- 失败或被标记异常的采集批次

每项至少包含稳定问题 ID、严重度、来源、数据批次、影响范围、数量、证据摘要、处理状态和导航目标。工作台展示优先项，审计页展示完整解释，设备数据页负责定位设备。

处理状态采用 `未处理 / 已确认 / 已忽略 / 已解决`。忽略必须记录原因；能够由后续数据自动证明恢复的问题应自动转为已解决，不要求人工重复操作。

## 6. 页面设计

### 6.1 工作台

首屏按行动优先组织，不采用大面积卡片堆叠：

1. 顶部状态栏：数据更新时间、数据新鲜度、采集健康度、当前批次和“开始采集”主操作。
2. 待处理表：严重度、来源、事项、范围、数量、更新时间和处理入口。
3. 紧凑指标带：总设备、开机、关机、离线、未知、需关注。
4. 楼栋状态表：支持按离线率、异常数和数据新鲜度排序。
5. 最近变化：相较上一完成批次新增离线、恢复在线和状态变化。

路径、规则版本和技术来源信息移到诊断页。空数据、数据过期、审计未运行和加载失败都必须给出可执行下一步。

### 6.2 采集

页面有三个互斥状态：

- 准备中：范围选择、任务模式和运行前检查。
- 运行中：阶段时间线、当前楼栋、采集计数、数据状态、运行时长、最近心跳和日志。
- 已完成：采集结果、导入结果、质量结论，以及“查看数据”“查看问题”“导出”的下一步。

默认界面只显示范围和任务模式。补采定位、实时批量大小、超时、重开间隔、最大设备数、自检、网络监听等进入“工程参数”折叠区。

运行前阻断项必须带直接操作，例如打开采集浏览器、重新检查或打开系统设置。新增“只重试失败楼栋”，不提供底层不支持的虚假暂停或断点续采。

任务运行期间锁定会影响执行口径的设置。停止操作要明确说明已完成阶段和当前数据是否已更新。

### 6.3 设备数据

设备数据保留“当前筛选结果即 Excel 导出口径”的产品不变量。

筛选改为可折叠面板，并分为：

- 位置：楼栋、座号、楼层、页面、设备名。
- 状态：开机、关机、离线、未知、模式、风速、温度、集控锁定。
- 业务范围：公区、非公区和自定义分组。

顶部始终展示已生效条件和结果数量。增加带数量的快捷筛选：离线、未知、温度异常、无实时数据和需关注。

新增以下能力：

- 自定义分组筛选和成员设备 Excel 导出。
- 保存、重命名和删除常用筛选视图。
- 右侧设备详情面板，展示基础信息、采集值、实时值、质量问题、关注事件和历史变化。
- 列排序、列显示配置和紧凑模式。
- 历史批次只读比较，展示相对当前批次的状态变化。
- 导出前显示将导出的设备数量；成功后提供打开文件和打开目录。

查询失败时保留筛选和已有表格，使用页面级错误条提供重试。刷新期间不清空已有结果。历史、过期和部分失败数据必须有持续可见的状态标识。

### 6.4 规则与计划

规则与计划包含四个内部视图：

1. 分组
2. 日期计划
3. 关注规则
4. 楼层目录（高级维护）

分组采用主从布局：左侧分组列表，中间成员表，右侧编辑面板。系统分组只读，自定义分组可维护名称、级别、说明、状态和成员。

成员维护支持单项选择、批量选择候选设备、粘贴设备名、冲突预览和批量移除。保存前必须明确新增、重复、冲突和无法匹配数量。

日期计划采用三栏布局：左侧区域组和计划组，中间月历，右侧所选日期的规则、时段和适用对象。支持跨月多选，但保存操作必须显示影响日期数和对象数。

关注规则独立展示适用分组、时间窗口、当前状态和异常事件。异常事件通过调查上下文精确进入设备数据页。

楼层候选目录属于高级数据维护，不得与分组成员操作混淆。

### 6.5 审计

审计页以统一问题工作台为核心：

- 顶部按严重度、来源、楼栋和处理状态筛选。
- 中间展示基础质量、实时点位、对账、计划偏差和关注事件。
- 右侧展示规则说明、证据、影响设备和建议操作。
- 提供定位设备、标记已知、暂时忽略和确认处理。

历史批次放入独立“版本”视图，支持选择两个批次进行摘要差异比较。恢复和删除属于高风险操作，放入溢出菜单并使用明确确认对话框。恢复前显示目标批次、当前数据备份策略和受影响记录数。

### 6.6 系统设置

设置页保留四组配置：EMS 连接、目录、采集默认值和外观。

新增：

- EMS 连接测试。
- 数据目录可写性、数据库状态和剩余空间检查。
- 未保存更改提示。
- 设置影响说明，例如“立即生效”“下次采集生效”或“需要重启”。

迁移旧数据移动到诊断维护区。执行前必须展示来源、目标、冲突和备份策略。

### 6.7 诊断

诊断页先展示环境健康摘要，再展示应用信息、路径、日志和最近 Excel。

环境检查覆盖：

- .NET 运行时和应用包身份
- Windows App SDK
- Sidecar 文件和启动能力
- Edge 和 CDP 可用性
- 数据库路径、schema 和迁移状态
- 日志与导出目录可写性

日志查看支持级别、类别和时间筛选，以及复制错误编号。诊断页可以复制经过脱敏的摘要文本，但不新增第二种业务报表或用户数据导出入口。

## 7. 交互与视觉规范

- 使用 WinUI 原生图标和现有主题资源，不引入独立图标体系。
- 工具按钮优先使用图标；不熟悉的图标必须有工具提示和自动化名称。
- 页面标题保持 24px 级别，紧凑面板标题使用现有 17px 级别，不使用营销式大标题。
- 状态颜色必须同时配合文字或图标，不单独依赖颜色表达。
- 正常状态保持克制，警告和错误颜色只用于需要行动的内容。
- 卡片只用于独立指标、工具或重复条目，不嵌套卡片，不把整页分区都做成浮动卡片。
- 表格和固定工具栏使用稳定列宽、最小宽度和滚动策略，动态内容不得推动整体布局跳动。
- 最低支持窗口为 1280x720；低于 1366px 宽时将筛选面板收为抽屉，1366px 及以上使用固定筛选栏，详情栏在可用宽度不足时同样切换为抽屉。
- 所有破坏性操作使用清晰文本按钮或溢出菜单，并在执行前说明具体对象和影响。

## 8. 数据流与状态

### 8.1 读取流程

数据上下文变化后，各页面通过共享服务读取同一批次。页面保留上一次成功数据，显示加载状态；只有新请求成功后才替换内容。过期响应不得覆盖较新的用户选择。

### 8.2 筛选与导出

设备查询和 Excel 导出必须接收同一个不可变查询对象。导出不得根据界面文字重新拼装条件。导出前的数量预览、页面结果总数和工作簿数据行数必须来自同一查询口径。

### 8.3 分组成员

分组成员解析为稳定设备身份集合。数据管理按分组筛选时，查询层使用分组 ID，不把大量设备名展开为界面参数。分组配置不做历史版本化；历史批次中的分组筛选使用当前分组成员集合，与所选历史批次的稳定设备身份求交集，并明确显示无法在该批次解析的成员数量。

### 8.4 问题状态

待处理问题保存稳定身份和状态变化记录。忽略、确认和解决操作不能修改原始采集证据。自动解决必须保留此前问题记录和解决批次。

## 9. 错误处理

- 应用启动迁移失败继续使用全局不可关闭错误条，提供重试和打开设置。
- 页面读取失败使用页面级错误条，保留当前上下文和可恢复数据。
- 操作失败显示稳定错误码、安全用户文案和重试入口，不直接呈现原始异常。
- 部分成功必须明确列出成功范围、失败范围和当前数据库是否已更新。
- 删除、恢复、迁移和批量移除均需要对象级确认；确认文案不得使用泛化的“是否继续”。
- 无数据、无筛选结果、尚未审计和权限/状态不允许属于不同空态，文案和操作必须区分。

## 10. 实施优先级

### P0：可运行与核心闭环

- 通过安装、启动和 MSIX 生命周期验证门。
- 收敛主导航和工具入口。
- 统一数据上下文栏和调查导航上下文。
- 建立待处理队列的最小可用模型。
- 增加设备快捷筛选、详情面板和稳定加载/空态/错误态。
- 保证筛选结果、导出预览和 Excel 行数一致。

### P1：跨页业务闭环

- 自定义分组筛选和成员导出。
- 拆分规则与计划页面。
- 只重试失败楼栋。
- 保存筛选视图。
- 历史批次差异比较。
- 审计问题处理状态和精确设备定位。

### P2：效率与深度

- 表格列个性化。
- 批量成员录入和冲突处理。
- 关注事件完整生命周期。
- 日志高级筛选。
- 更广泛的分辨率、高 DPI 和无障碍优化。

P0 完成并通过验收后才能进入 P1；P1 的数据模型必须在实施计划中进一步拆分，避免一次同时修改导航、查询、schema 和所有页面。

## 11. 测试与验收

### 11.1 自动化

- 导航合同：5 个主入口、2 个底部工具入口和深链接目标正确。
- 数据上下文合同：当前/历史状态同步，历史写操作和导出被禁止。
- 查询合同：快捷筛选、自定义分组、保存视图和历史比较使用明确查询模型。
- 导出合同：结果总数、导出预览和 Excel 数据行数一致。
- 待处理队列合同：各来源映射、稳定问题 ID、状态转换和自动解决可重复验证。
- ViewModel 并发测试：较旧异步响应不能覆盖较新选择。
- UI 合同：错误、空态、加载态、只读态和破坏性操作均存在明确入口与文案。

### 11.2 Windows 运行态

- 1280x720、1366x768 和 1920x1080。
- 100% 和 150% 缩放。
- 浅色、深色和跟随系统主题。
- 键盘导航、焦点可见、屏幕阅读器名称和高对比度检查。
- 长设备名、长路径、最大筛选数量和无数据场景不重叠、不截断关键操作。
- MSIX 安装、首次启动、升级、卸载和重新安装。
- 安装包内 Sidecar 启动及隔离现场 E2E。

真实 EMS 现场测试与本地合成数据测试必须分别记录；本地通过不得描述为真实 EMS 端到端通过。

## 12. 非目标

- 不增加空调远程控制。
- 不引入账号、角色和权限系统。
- 不恢复 legacy Web 面板为产品入口。
- 不增加 TXT、Markdown 或多格式业务报表。
- 不改变“设备数据当前筛选 Excel”为唯一用户数据导出方式。
- 不在本设计阶段修改采集规则或通讯状态判定。

## 13. 实施边界

本规格覆盖完整产品方向，但实施应拆成独立计划：

1. P0-A：Windows 启动与运行态 UI 验证。
2. P0-B：导航、数据上下文和调查导航。
3. P0-C：待处理队列与工作台。
4. P0-D：设备数据筛选、详情和导出一致性。
5. P1-A：规则与计划重组及分组筛选。
6. P1-B：审计处理状态与历史比较。
7. P2：效率、诊断和无障碍增强。

每个实施计划都必须保持生产数据库安全不变量，并使用临时数据库完成 schema、导入、审计和 Excel 验证。
