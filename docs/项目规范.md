# EMS Scout 项目规范

更新时间：2026-07-11

本文是 EMS Scout 的开发和交付规范。架构实现以 `docs/architecture.md` 为准，
当前完成度以 `docs/状态.md` 为准，Windows 验收按 `docs/Windows验证清单.md` 执行。

## 1. 产品与命名

- 面向用户的产品名称统一写作 `EMS Scout`。
- C# 程序集、命名空间、项目目录和类型前缀统一写作 `EmsScout`。
- npm 包名受小写规则约束，使用 `emsscout`。
- 契约 URI、数据库字段、环境变量和历史文件名属于稳定机器标识，不因品牌调整直接改名。
- `AC-Scout.bat` 是受保护的 legacy 兼容入口；文件名保留，显示文案使用 `EMS Scout`。
- 新增用户界面、日志标题、安装包显示名或文档标题时，不得引入新的产品别名。

## 2. 产品范围

EMS Scout 的主流程固定为：

```text
EMS 页面采集 -> CollectionSnapshot -> SQLite -> 原生质量/对账
             -> 数据管理筛选 -> 导出当前筛选 Excel
```

唯一面向用户的导出方式是：

`数据管理 -> 导出当前筛选 Excel`

TXT、Markdown、多格式报表、旧 Web/Electron 面板和 TUI 不属于当前产品主入口。
在 Windows 安装包和真实 EMS 验收完成前，它们按 `docs/legacy-inventory.md` 保留为
受保护兼容路径。

## 3. 语言和所有权

| 技术 | 必须负责 | 不得负责 |
|---|---|---|
| C# / .NET 10 | WinUI、契约校验、SQLite 生命周期、质量、对账、筛选、Excel、错误和日志 | EMS DOM/Shadow DOM 采集 |
| Node.js | Playwright + Edge CDP 采集、Sidecar 协议适配 | 产品 UI、数据库迁移、主质量审计、用户导出 |
| PowerShell | Windows x64 打包、隔离 Edge、现场 E2E、生产文件守护 | 产品业务规则 |
| SQL | 版本化 schema 资源和查询 | 仓储运行时静默建表或修表 |

只有 `EmsScout.Infrastructure/Migrations` 可以执行 `CREATE TABLE` 或
`ALTER TABLE`。仓储只能验证 schema 并明确失败，不能在用户操作中修改结构。

## 4. 分层和依赖

```text
EmsScout.Desktop
  -> EmsScout.Application
  -> EmsScout.Domain
  -> EmsScout.Infrastructure
```

- `Domain` 保存稳定业务模型，不依赖 UI、SQLite 或 Playwright。
- `Application` 定义用例、端口、工作流、错误和展示契约。
- `Infrastructure` 实现 SQLite、文件、Excel、Sidecar 和操作系统适配。
- `Desktop` 负责 WinUI 页面、绑定和交互，不实现数据库 DDL 或采集协议。
- Node 采集器通过版本化契约跨进程通信，不直接操作产品数据库。

新增依赖方向或跨层调用前，必须先更新 ADR 或 `docs/architecture.md`。

## 5. 契约和兼容性

- 进程边界只使用 `contracts/` 中有版本号的 JSON Schema。
- 当前稳定契约为 `CollectionSnapshot v1`、`WorkflowEvent v1`、`WorkflowControl v1`。
- Sidecar stdout 只能写 WorkflowEvent NDJSON；人类日志写 stderr。
- 每个工作流必须恰好有一个 terminal event。
- 现有 v1 字段不得静默改变含义、类型或枚举值。
- 破坏性变更必须新增契约版本、fixture、双端解析测试和迁移说明。
- `IND_MAP` 是来源事实：绿色关机、红色开机、灰色离线；缺失状态不得猜测。

## 6. 数据库和身份

- SQLite 迁移必须版本化、可审计，并在事务中执行。
- 导入必须支持取消、失败回滚、幂等和用户数据保全。
- `source_key` 表示来源位置，`device_uid` 表示稳定设备身份。
- 身份歧义必须写入 alias/ambiguity ledger，不得猜测绑定。
- 质量标签只是证据；原生质量层必须根据卡片内容重新审计。
- `2.5F`、`B1F`、`BM` 都是合法楼层标签，不得强制转成整数楼层。

## 7. 生产数据安全

以下规则没有例外：

- 测试不得通过 SQLite 打开生产 `out/ac.db`。
- 需要生产基线时，先按字节复制 DB 和非空 WAL 到系统临时目录，再查询副本。
- 不删除、清理、暂存或改写 `data/1号楼`、`data/2号楼` 的 WAL/SHM 证据。
- 现场 E2E 只能写唯一 `out/field-e2e-*` 目录。
- 现场导入必须显式传临时 snapshot、DB、质量报告和 Excel 路径。
- 任何路径解析到生产 `out/ac.db` 时必须拒绝执行。
- 验证前后应比较生产 JSON、DB、WAL、SHM 的大小和 SHA-256。
- 本地临时 DB/Excel smoke 不得描述为真实 EMS E2E 通过。

`out/` 不进入 Git。依赖 run17 或现场实时文件的测试统一标记：

```text
Fixture=ProductionEvidence
```

干净克隆默认排除该组；只有安全放置完整证据后才运行它。

## 8. 代码规范

- 优先沿用现有模块和命名，不为单一调用新增抽象层。
- 新逻辑必须有与风险相称的测试；共享规则必须有跨语言 fixture 或黄金测试。
- JavaScript 使用 CommonJS、`'use strict'` 和明确的输入/输出边界。
- C# 启用 nullable，异步 API 传递 `CancellationToken`，异常在边界统一分类。
- PowerShell 脚本设置 `$ErrorActionPreference = 'Stop'`，外部进程必须检查退出码。
- SQL 必须参数化；禁止把用户输入拼接进 SQL。
- 日志使用结构化事件、稳定错误码和脱敏字段，不直接记录 token、密码或敏感 URL 参数。
- 注释解释不明显的约束或原因，不复述代码。
- 当前源码以 ASCII 为默认；已有中文 UI 和中文文档可继续使用 UTF-8。

## 9. 测试分层

### 干净克隆门禁

```powershell
npm ci
dotnet restore native\EmsScout.Native.slnx -r win-x64
node --test sidecar/test/*.test.js tests/architecture/*.test.js tests/contract-audit/*.test.js tests/enumeration/*.test.js tests/field-e2e/*.test.js tests/golden/*.test.js
npm run self-test
npm run native:test
dotnet format native\EmsScout.Native.slnx --verify-no-changes --no-restore
```

### 生产证据门禁

仅在本机安全放置 `out/` 证据后运行：

```powershell
npm run native:test:evidence
```

### Windows 安装包和现场门禁

按 `docs/Windows验证清单.md` 执行。Windows XAML、MSIX、内置 Sidecar 和真实 EMS
任何一项未验证时，都不得声明最终交付完成。

## 10. Git 和提交

- 不提交 `out/`、`node_modules/`、`artifacts/`、`bin/`、`obj/`、日志或临时文件。
- 不使用 `git add -A`；按明确文件组暂存，并在提交前运行 `git diff --cached --check`。
- 不回退或覆盖来源不明的工作区改动。
- 提交信息使用简洁的 Conventional Commit 风格：`feat:`、`fix:`、`refactor:`、
  `test:`、`build:`、`docs:`、`chore:`。
- 一个提交只表达一个可审查目的；代码、测试和必要文档应在同一提交中保持一致。
- push 前必须确认分支、远端、测试结果和受保护文件状态。
- 未经明确授权，不改生产数据、不删除 protected legacy、不创建发布或 tag。

## 11. 文档维护

行为或所有权改变时，必须同步更新：

- 架构边界：`docs/architecture.md` 或新 ADR。
- 当前完成度和外部门：`docs/状态.md`。
- 下一台机器操作：`docs/交接.md`、`docs/Windows验证清单.md`。
- 兼容代码处置：`docs/legacy-inventory.md`。
- 用户入口和常用命令：`README.md`。
- 重要变更：`CHANGELOG.md`。

历史记录可以保留旧名称和旧命令，但当前规范、当前 UI 和当前入口不得使用过期品牌。

## 12. 完成定义

一项工作只有同时满足以下条件才算完成：

1. 需求行为已经实现，不以更窄的替代行为代替。
2. 数据和兼容性边界保持成立。
3. 相关自动化测试通过，无法执行的门禁被明确列为未验证。
4. 当前文档与代码状态一致。
5. `git diff --check` 通过，没有误提交运行产物或生产证据。
6. Windows/现场事项有可追溯的命令、日志和隔离输出目录。
